# 🐞 发现的错误和漏洞汇总报告

**目标：** 整理来自静态分析、业务逻辑审查、安全审查和单元测试的所有问题，并分类。

## 📋 1. 静态分析和代码风格问题 (Refactoring/Linting)

*   **文件：** `utils/helpers.py`
    *   **问题：** 函数 `calculate_hash` 的类型提示缺失，应明确输入和输出类型。
    *   **建议：** 添加类型提示，提高代码可读性和可维护性。
*   **文件：** `app.py`
    *   **问题：** 缺少对全局配置变量的初始化检查，如果 `config.py` 未正确加载，可能导致运行时错误。
    *   **建议：** 在应用启动时增加配置校验逻辑。
*   **文件：** `models/task.py`
    *   **问题：** 某些字段（如 `due_date`）的默认值处理不够健壮，如果传入的日期格式不正确，可能导致ORM层报错。
    *   **建议：** 在模型层增加日期格式校验或使用更严格的类型定义。

## 🐛 2. 业务逻辑和数据模型问题 (Bug)

*   **文件：** `api/tasks.py` (任务创建/更新)
    *   **问题：** **任务状态流转逻辑缺陷：** 当用户尝试将任务从“已完成”状态修改回其他状态时，业务逻辑没有进行阻止，导致数据不一致。
    *   **影响：** 任务状态的完整性被破坏。
    *   **修复点：** 在任务更新的业务函数中，增加状态校验，确保“已完成”状态不可逆转。
*   **文件：** `api/users.py` (用户管理)
    *   **问题：** **权限检查缺失：** 在获取用户详情的 API 接口中，没有检查当前登录用户是否拥有查看目标用户信息的权限（例如，普通用户是否可以查看其他用户的敏感信息）。
    *   **影响：** 存在越权访问的风险。
    *   **修复点：** 在 `get_user_details` 函数中，增加权限校验逻辑。
*   **文件：** `utils/database.py`
    *   **问题：** **数据库连接资源管理：** 在使用数据库连接时，没有使用 `with` 语句或确保连接在使用完毕后被正确关闭，可能导致连接池资源泄露。
    *   **影响：** 在高并发场景下，可能耗尽数据库连接资源。

## 🔒 3. 安全漏洞问题 (Security)

*   **文件：** `api/tasks.py` (任务查询)
    *   **问题：** **潜在的注入风险 (SQL/NoSQL)：** 虽然使用了 ORM，但在构建复杂的查询条件时，如果直接拼接用户输入（例如，在 `WHERE` 子句中），仍存在潜在的注入风险。
    *   **建议：** 确保所有用户输入都经过参数化查询或严格的输入验证。
*   **文件：** `api/users.py` (用户认证)
    *   **问题：** **密码哈希处理不当：** 虽然使用了哈希，但如果哈希算法版本过旧或盐值（salt）处理不当，可能存在密码破解的风险。
    *   **建议：** 升级到更现代、更安全的哈希算法（如 Argon2 或 Bcrypt），并确保盐值是随机且唯一的。
*   **文件：** `config.py`
    *   **问题：** **敏感信息硬编码：** 配置文件中可能存在测试环境或开发环境的敏感密钥（如 API Key, Secret Key）被硬编码，这在部署到生产环境时极不安全。
    *   **建议：** 所有敏感配置应通过环境变量（Environment Variables）加载，而不是直接写在代码中。

## 🧹 4. 重构和优化建议 (Refactoring)

*   **文件：** `app.py`
    *   **问题：** **代码耦合度高：** `app.py` 承担了初始化、路由注册、配置加载等多个职责，导致职责单一性差。
    *   **建议：** 考虑将应用初始化和配置加载逻辑封装到单独的 `app_factory.py` 或 `main.py` 中，提高模块化程度。
*   **文件：** `utils/helpers.py`
    *   **问题：** **重复代码 (DRY)：** 多个地方可能重复编写了相似的日期格式化或验证逻辑。
    *   **建议：** 将这些通用工具函数提取到专门的 `utils/validation.py` 或 `utils/date_utils.py` 中，避免重复。

--- 

**总结：**

本次审查发现的问题涵盖了从基础的类型错误到高风险的安全漏洞，需要分阶段修复。优先处理 **Security** 和 **Bug** 类别的问题，特别是密码哈希和越权访问的修复。
